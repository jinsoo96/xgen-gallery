**한 줄 요약** — 컴파일의 본질은 코드 변환이 아니라 암묵 상태의 동결입니다. 전역 레지스트리를 설정 직렬화로 바꿔서, 산출물이 플랫폼 없이 환경변수만으로 동작하게 만들었습니다.

하네스의 승부수는 컴파일입니다. 캔버스에서 만든 워크플로우를 `compile_workflow()`로 pip wheel이나 npm 패키지로 변환하면, 플랫폼 없이 **환경변수만 주입하면 동작하는** 독립 산출물이 되고, MCP 서버로 기동해 Claude 같은 외부 에이전트의 도구가 됩니다. 이 기능을 완성하며 배운 것은, 컴파일의 어려움이 코드 변환이 아니라 **상태의 완전한 동결(freeze)**에 있다는 사실이었습니다.

## 원칙: 실행에 필요한 모든 것은 산출물 안에 있어야 합니다

첫 검증에서 산출물의 도구 정의가 빈 배열(`FROZEN_TOOL_DEFINITIONS = []`)로 생성되는 결함이 나왔습니다. freeze 빌더에 "워크플로우를 도구로 변환하는" 분기가 없었기 때문입니다. 해법은 유형별 동결 규칙을 채우는 것이었습니다 — 서브워크플로우는 설정 전체를 wheel에 동결해 포함하고 호출 시 중첩 파이프라인으로 in-process 실행(재귀 깊이 가드 포함), 캔버스는 하네스 설정으로 변환(transpile)하거나 다중 포트 DAG 인터프리터로 실행. Python과 Node 두 실행 채널이 같은 동결 포맷을 읽도록 패리티를 맞췄습니다.

## 반복 발생한 결함의 공통 구조: 프로세스 전역 상태

이후 몇 주간의 결함들은 패턴이 같았습니다. **클러스터 안에서는 정상인데 산출물에서는 조용히 기능이 빠지는** 부류입니다.

- judge 평가축 정의가 프로세스 전역 레지스트리에 있었습니다 — 산출물(새 프로세스)의 레지스트리는 비어 있으니 평가는 generic 폴백으로 무력화됩니다. 수정은 평가축을 **설정 객체에 직렬화**해 산출물이 자기 안에서 복원하게 하는 것.
- 문서 검색(RAG)은 서비스 참조·top_k 기본값·결과 미리보기 길이 세 곳이 동시에 산출물 경로에서 비활성 상태였습니다. 각각 동결 시점에 값을 확정해 포함하도록 바꿨습니다.

일반화하면 이렇습니다.

> in-process에서만 성립하는 암묵 상태(전역 레지스트리, 서비스 참조, 런타임 기본값)는 컴파일의 적입니다. 산출물이 읽을 수 있는 것은 직렬화된 설정뿐이므로, "동결 시점에 모든 암묵 상태를 명시 상태로 바꾼다"가 컴파일러의 첫 번째 계약이어야 합니다.

이 부류는 조용히 실패한다는 점이 특히 까다로워서, 검증도 산출물 기준으로 바꿨습니다 — 실제 Claude 클라이언트에 MCP로 등록해 도구 호출·데이터 인용까지 전 구간을 완주시키는 것을 완성 조건으로 삼았고, OAuth 인증과 표준 디스커버리 경로까지 포함해 검증 매트릭스 여섯 항목을 통과시킨 뒤에야 기능을 닫았습니다.

## 정리하며

컴파일 기능의 설계 원칙은 한 줄로 요약됩니다 — **산출물의 실행 경로에서 암묵 상태를 없앨 것.** 그리고 그 검증은 산출물 프로세스에서 해야 한다는 것이 우리가 얻은 결론입니다. in-process 테스트로는 이 부류의 결함이 구조적으로 잡히지 않았습니다. 다음 편은 실행 품질 쪽의 같은 질문입니다 — 에이전트가 지켜야 할 규칙을, 프롬프트가 아니라 어디에 두어야 하는가.

**같은 문제를 겪는 팀에게** — 우리가 겪으며 얻은 생각들입니다.
- 컴파일·산출물 기능이라면 "동결 시점에 암묵 상태를 명시 상태로"라는 규칙을 먼저 세워 두는 것도 좋을 것 같습니다. 우리 경우 전역 레지스트리, 서비스 참조, 런타임 기본값 — 이 세 가지가 주된 원인이었습니다.
- 산출물 검증은 별도 프로세스에서 실제 클라이언트(MCP 등록→도구 호출→데이터 인용)로 하는 편이 확실했습니다. in-process 테스트로는 이 부류의 결함이 잡히지 않았습니다.
