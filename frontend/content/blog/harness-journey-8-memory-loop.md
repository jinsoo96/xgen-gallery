**한 줄 요약** — 답변 직후(s09) 훅이 기억을 추출해 4개 스코프로 라우팅하고, 다음 실행의 프롬프트 재조립(s03)이 그것을 다시 읽습니다. 이렇게 해서 실행 경계를 넘는 학습 루프가 닫혔습니다. 이유는 단순합니다 — 저장만 하고 읽지 않는 기억은 기억이 아니기 때문입니다.

앞 편의 자가단조는 실행과 실행 사이에서 설정을 진화시킵니다. 그런데 설정 말고 **경험**은 어떻게 되는가. 에이전트 메모리의 산업 지형을 조사(출처 24건, 클레임 119건을 3인 교차검증해 만장일치만 수록)하고 내린 우리 시스템의 진단은 한 문장이었습니다.

> "실행 내 루프(재시도·자기교정)는 있는데, 실행 간 루프가 끊겨 있다."

## 어디서 뽑고, 어디에 넣는가

배선의 핵심은 파이프라인의 두 지점입니다. **추출은 s09(마무리 스테이지)의 훅에서** — 실행이 끝나는 시점에 대화에서 기억할 것을 뽑아 저장합니다. **주입은 s03(프롬프트 재조립)에서** — 다음 실행이 시작될 때 회상된 기억을 컨텍스트에 넣습니다. 라이브 검증에서 이 두 지점의 결함이 각각 하나씩 나왔는데, 추출 훅이 특정 저장 전략에서만 동작하던 것과 회상된 기억이 재조립 단계에서 유실되던 것이었습니다. 둘 다 "지점이 명시된 배선"이었기에 특정과 수정이 빨랐습니다.

첫 종단 실측: 한 대화에서 "제 이름은 김진수입니다"라고 말하면 s09가 기억 2건을 추출해 저장하고, **완전히 새로운 실행**에서 이름을 묻자 회상이 동작했습니다. 대화와 워크플로우의 경계를 넘는 첫 회상입니다.

## 스코프 4개와 우선순위 2축

기억을 한 통에 넣으면 회수가 오염됩니다. 그래서 저장소를 **session / workflow / user / platform** 4개 스코프로 나누고, 추출 시점에 LLM이 스코프를 라우팅합니다(라이브 검증: 한 발화에서 `user 1건, workflow 1건`으로 분기 저장). 충돌 해소는 2축입니다 —

- **타입 축**: 제약(constraint)은 상위 스코프가 우선합니다. 플랫폼 정책이 개인 취향을 이깁니다.
- **스코프 축**: 선호(preference)는 하위 스코프가 우선합니다. 이 워크플로우에서의 선호가 일반 선호를 이깁니다.

회수 설계에는 도구 쪽에서 검증된 패턴을 그대로 옮겼습니다 — **"도구의 점진 공개(PD)와 메모리의 점진 공개는 같은 문제"**라는 관찰입니다. 기억 전문을 다 싣지 않고 목차를 먼저 검색한 뒤 필요한 본문만 펼칩니다. 컨텍스트 예산 관점에서 도구 카탈로그와 기억 저장소는 동형이기 때문입니다.

## 교훈의 이월: 저장에서 소비까지

실패에서 얻은 교훈(lesson)도 같은 원칙을 따릅니다. 초기 구현은 교훈을 기록만 하고 다음 실행이 읽지 않았습니다 — 이 결선을 이어 Run A의 교훈이 Run B 시작 시 `[state] resume — 정제기억 61건·교훈 69건 이월`로 주입되는 것을 실측했습니다. 판정을 0/1 양극단에서 0.70~0.98의 점진 채점으로 바꾼 것도 같은 맥락입니다 — 이월할 신호의 해상도가 높아야 다음 실행의 조정이 정밀해집니다.

## 정리하며

메모리 기능의 성패는 저장소 스키마가 아니라 **파이프라인의 어느 지점에서 뽑고 어느 지점에서 읽는가**에 있었습니다. 추출은 s09, 주입은 s03, 라우팅은 4스코프, 충돌은 2축 — 전부 명시된 배선이라 검증도 지점 단위로 됩니다. 남은 과제는 기억 정리(중복 병합·감쇠) 루틴의 상시화입니다. 다음 편은 마지막 — 에이전트의 상황 인지 설계에서 아직 풀리지 않은 과제들입니다.

**같은 문제를 겪는 팀에게** — 우리가 겪으며 얻은 생각들입니다.
- 메모리는 저장보다 소비부터 설계하는 편이 좋았습니다. "다음 실행의 어느 지점이 이것을 읽는가"에 답이 없으면 쓰이지 않는 데이터로 남았습니다.
- 스코프 분리와 충돌 규칙(제약은 상위 우선, 선호는 하위 우선)의 명문화도 고려해 볼 만하다고 생각합니다. 한 통 저장소는 회수를 오염시켰습니다.
- 기억 회수에는 도구 PD 패턴(목차 검색 → 본문 펼침)을 그대로 재사용할 수 있었습니다. 컨텍스트 예산 문제로서 둘은 동형이었습니다.
