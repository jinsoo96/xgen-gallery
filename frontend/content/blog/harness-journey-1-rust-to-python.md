**한 줄 요약** — 에이전트 실행기의 병목은 CPU가 아니라 변경 속도입니다. 상태 머신과 체크리스트 스테이지, 그리고 Rust 대신 Python을 택한 이유를 정리합니다.

LLM 에이전트는 유능하지만, 실행을 시스템이 붙잡아 주지 않으면 품질이 요동칩니다. 우리가 "하네스(Harness)"라고 부르는 에이전트 실행기는 그 요동을 아키텍처로 제어하려는 시도입니다. 이 시리즈는 석 달간 세 저장소에 쌓인 1,270여 커밋에서, 우리가 부딪힌 기술 문제와 그것을 푼 설계를 골라 정리한 기록입니다.

## 문제 정의: DAG 실행기로는 에이전트를 붙잡을 수 없었습니다

출발점의 문제의식은 첫 README에 정리돼 있습니다. 기존 워크플로우 실행기는 LangGraph 기반 DAG여서 **사이클(재시도 루프)을 표현할 수 없고**, 100개가 넘는 제네릭 노드와 bool 토글로 파이프라인이 조립되어 실행 파이프라인 자체를 제어할 수 없었습니다. 에이전트에게 필요한 것은 "노드의 나열"이 아니라 **상태 머신** — 도구 탐색, 컨텍스트 압축, 검증, 재시도 판단이 명시적 스테이지로 존재하고, 스테이지를 넣다 뺐다 할 수 있는 구조였습니다.

그래서 첫 구현부터 스테이지는 체크리스트였습니다. 첫 커밋의 스테이지 enum 주석에 이미 "사용자가 체크리스트로 선택 — 포함된 단계만 실행됨"이 명시돼 있었고, minimal(4단계)부터 full(12단계)까지 프리셋이 함께 있었습니다. 컨텍스트 압축 예산 같은 상수는 공개된 코딩 에이전트 CLI 파이프라인을 분석해 대응표를 만들어 이식하되, 검증·판정 게이트는 "이식이 아니라 자체 설계"라고 처음부터 경계를 그었습니다.

## 설계 판단: Rust에서 Python으로

첫 커밋은 56파일 14,034줄의 Rust 상태 머신이었습니다. 나흘 뒤 이 코어를 전량 걷어내고 Pure Python으로 재작성했습니다. 되돌아보면 이 전환의 논리는 명확합니다.

> 에이전트 실행기의 병목은 CPU가 아니라 API 호출 대기입니다. 반면 요구사항의 중심은 "파이프라인 단계를 수시로 재조립하는 것"이었습니다. 최적화해야 할 것은 실행 속도가 아니라 **변경 속도**였습니다.

Rust가 주는 이점(성능·타입 안전)은 이 워크로드에서 실익이 크지 않았던 반면, 비용(Python 생태계와의 경계, PyO3 바인딩 유지비)은 실질적이었습니다. 실제로 PyO3 바인딩은 하루 만에 제거됐고, Rust 코드는 사라졌지만 스테이지 이름과 상태 머신 계약은 Python 구조에 그대로 승계됐습니다. 언어를 바꾼 것이지 설계를 버린 것이 아닙니다.

## 같은 날 세 갈래가 출발했습니다

Python 전환일에 엔진(실행기 본체), 이식(플랫폼 연결 레이어), UI(파이프라인 시각화 +3,694줄)가 각자의 저장소에서 동시에 시작됐습니다. 이틀 뒤 PyPI에 v0.1.0이 올라갔고, 같은 날 플랫폼 저장소는 내장 사본을 제거하고 PyPI 의존으로 전환했습니다 — 엔진과 플랫폼을 패키지 경계로 떼어 놓는 구조가 첫 주에 잡힌 것입니다. 이 분리가 왜 중요한지는 다음 편의 주제입니다.

## 정리하며

기술 선택의 기준은 "무엇이 좋은 기술인가"가 아니라 "이 시스템의 병목이 어디인가"였습니다. 에이전트 실행기의 병목은 재조립 속도였고, 그 기준이 상태 머신·체크리스트 스테이지·Python이라는 세 가지 선택을 일관되게 설명합니다. 다음 편은 이 실행기가 플랫폼과 만나는 방식 — "엔진은 플랫폼을 몰라야 한다"는 분리 원칙이 어떻게 코드로 강제됐는지입니다.

**같은 문제를 겪는 팀에게** — 우리가 겪으며 얻은 생각들입니다.
- 실행기를 고를 때 처리량보다 파이프라인 변경 빈도를 먼저 재 보는 것도 좋을 것 같습니다. 우리 경우 병목이 API 대기라서 언어 수준의 성능 이점은 실효가 크지 않았습니다.
- 스테이지를 enum+프리셋(체크리스트)으로 계약화해 두니, 이후의 재조립·컴파일·자가개선이 전부 그 계약 위에서 성립했습니다. 고려해 볼 만한 구조라고 생각합니다.
