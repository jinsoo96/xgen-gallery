**한 줄 요약** — 소스와 실제 배포된 산출물이 달라지는 원인은 추적 누락, 캐시 재사용, 데이터 계약 부재, 설정값 검증 부족이었습니다. 우리는 각각을 별도의 검증 단계로 막았습니다. 문제는 코드가 아니라 배포 과정에 있었기 때문입니다.

코드는 수정했습니다. 테스트도 모두 통과했습니다. CI도 성공했습니다. 그런데 사용자 환경에서는 여전히 문제가 발생했습니다.

처음에는 원인을 이해하기 어려웠습니다. 같은 소스에서 빌드했는데, 왜 사용자가 설치한 패키지는 다른 동작을 하는 걸까요?

몇 번의 릴리즈를 반복하면서 한 가지 사실을 알게 됐습니다. 배포는 코드를 전달하는 과정이 아니라, 코드가 변하지 않았음을 증명하는 과정이라는 점입니다. 이번 글은 그 과정에서 발견한 문제와, 배포를 신뢰할 수 있도록 만들기 위해 추가한 검증 단계를 소개합니다.

## 문제는 코드가 아니라 배포 과정에 있었습니다

7주 동안 PyPI와 npm에 30회가 넘는 릴리즈를 진행했습니다. 흥미로운 점은 대부분의 장애가 코드에서 발생하지 않았다는 것입니다.

오히려 저장소에 포함되지 않은 파일, CI 캐시, 중복된 데이터, 잘못된 설정값처럼 배포 과정에서만 드러나는 문제가 더 많았습니다.

처음에는 서로 다른 버그처럼 보였습니다. 하지만 하나씩 원인을 추적하면서 공통점을 발견했습니다. 소스와 실제 산출물 사이에는 생각보다 많은 변수가 존재했습니다.

## 로컬에서 성공했다고 배포가 성공한 것은 아닙니다

가장 먼저 만난 문제는 .gitignore였습니다. 임시 파일을 제외하기 위해 추가한 규칙 하나가 __init__.py까지 제외하고 있었습니다.

개발 환경에서는 아무 문제가 없었습니다. 이미 로컬 디스크에는 해당 파일이 존재했기 때문입니다. 하지만 새로운 환경에서 저장소를 다시 내려받아 빌드하면 상황이 달랐습니다. 핵심 파일이 빠진 상태로 패키지가 생성됐고, 그 결과 네 번의 릴리즈가 연속으로 잘못 배포됐습니다.

이 경험 이후 배포 기준을 바꿨습니다. 배포는 로컬 환경이 아니라 Fresh Clone 환경에서 검증하기 시작했습니다. 로컬 빌드가 성공했다는 것은 내 컴퓨터를 증명할 뿐, 저장소를 증명하지는 못하기 때문입니다.

## 배포는 끝난 뒤에도 실패할 수 있습니다

패키지를 정상적으로 발행했다고 해서 끝나는 것도 아니었습니다. CI는 이전 Docker 레이어를 재사용했고, 동시에 실행된 두 개의 워크플로우는 같은 버전 번호를 발행하기도 했습니다. 결과적으로 사용자가 설치한 패키지는 우리가 의도한 코드와 달랐습니다.

이후에는 배포 절차를 다시 설계했습니다. 발행 전에 원격 저장소 상태를 확인하고, 배포가 끝난 뒤에는 실제 패키지를 다시 다운로드해 내용까지 검증했습니다.

PyPI는 한 번 배포하면 되돌릴 수 없습니다. 그래서 마지막 검증 대상은 CI 결과가 아니라 실제 사용자가 설치하는 패키지가 되었습니다.

## 데이터도 계약이 없으면 예측할 수 없습니다

문제는 패키지에만 있지 않았습니다. 도구 목록에는 존재하는데 호출하면 404가 발생하는 현상도 있었습니다.

원인은 의외였습니다. 워크플로우 ID가 유일하지 않았습니다. 발행된 버전과 오래된 초안이 같은 식별자를 사용했고, 목록 조회와 실제 실행이 서로 다른 데이터를 참조하고 있었습니다. 또 다른 문제에서는 동일한 API가 두 위치에 등록돼 있었습니다.

결국 모두 같은 문제였습니다. 시스템은 식별자가 하나라고 믿고 있었지만, 데이터는 그렇지 않았습니다. 이후에는 UNIQUE 제약을 추가하고, 등록 지점 역시 하나로 통합했습니다.

## 설정 하나가 시스템 전체를 멈출 수도 있습니다

사용자가 설정한 max_tokens 값 하나 때문에 모든 호출이 실패한 적도 있었습니다. 노드에는 100만 토큰이 설정되어 있었지만, 실제 모델은 12만 8천 토큰까지만 처리할 수 있었습니다. 실행기는 이를 알지 못한 채 같은 요청을 90번 반복했고, 결국 비용만 소비한 뒤 실패했습니다.

이 문제 이후에는 설정도 검증 대상이 됐습니다. 사용자가 입력한 값은 모델이 허용하는 범위 안으로 자동 보정하고, 반복 실행에서는 의미 있는 진전이 없는 루프를 감지하도록 변경했습니다. 설정 역시 신뢰해서는 안 되는 입력이라는 사실을 배운 순간이었습니다.

## 검증 단계는 계속 늘어났습니다

돌이켜보면 모든 문제의 원인은 달랐습니다. 하지만 해결 방법은 하나였습니다. 검증 단계를 추가하는 것.

Fresh Clone에서 다시 빌드하고, 원격 저장소를 확인하고, 실제 패키지를 다시 설치하고, 독립된 실행 환경에서 End-to-End 테스트를 수행했습니다.

배포 파이프라인은 점점 길어졌습니다. 하지만 릴리즈는 오히려 더 안정적으로 변했습니다. 검증이 늘어난 것이 아니라, 불확실성이 줄어든 것이었습니다.

## 마치며

이번 경험을 통해 얻은 결론은 단순했습니다. 배포는 코드를 복사하는 과정이 아닙니다. 사용자가 실행하는 코드가 개발자가 의도한 코드와 동일하다는 것을 증명하는 과정입니다.

그래서 신뢰는 빌드에서 만들어지지 않았습니다. 검증 단계에서 만들어졌습니다. 하네스는 코드보다 검증 파이프라인을 먼저 신뢰하는 방향으로 발전했습니다.

다음 글에서는 이렇게 안정적으로 배포된 실행기를 기반으로, 모델보다 설정이 실행 품질에 얼마나 큰 영향을 주는지 실험한 내용을 소개합니다.

**같은 고민을 하고 있다면** — 배포 검증은 로컬 환경이 아니라 Fresh Clone을 기준으로 시작하는 것이 좋습니다.
- 실제 패키지를 다시 다운로드해 검증하는 절차까지 포함해야 개발자가 본 코드와 사용자가 설치하는 코드가 같은지 확인할 수 있습니다.
- 데이터 계약과 사용자 설정 역시 배포 파이프라인의 일부로 생각해야 합니다. 식별자의 유일성, 등록 지점의 단일성, 실행 환경의 한계를 고려한 설정 검증은 코드 수정만으로는 해결할 수 없는 운영 문제를 크게 줄여줍니다.
