## 경로 1: 추적 누락 — 저장소가 소스의 전부가 아닐 때

가장 값비싼 사례는 `.gitignore`였습니다. 스크래치 파일용 `_*.py` 패턴이 `__init__.py`까지 매칭해, 새 패키지의 초기화 파일이 커밋에서 계속 빠졌습니다. 로컬 빌드는 디스크의 untracked 파일 덕에 정상이지만, fresh-clone에서 빌드된 wheel은 패키지 구조가 깨진 채 네 개 버전이 연속 배포됐습니다. 같은 패턴이 테스트 디렉토리도 막고 있었습니다 — 테스트 부재의 원인이 작성이 아니라 추적이었던 것입니다(해제 후 0→237개로 성장).

방어층: **배포 검증은 반드시 fresh-clone 기준으로.** 로컬 빌드 성공은 "디스크에 있는 파일"의 증명일 뿐 "저장소에 있는 파일"의 증명이 아닙니다.

## 경로 2: 캐시와 경합 — 발행 이후에도 어긋납니다

핫픽스를 내도 CI의 도커 레이어 캐시가 깨진 이전 버전을 재사용할 수 있고(캐시 무효화를 복구 절차에 포함), 두 작업 선이 같은 버전 번호로 동시에 발행하면 한쪽 수정이 빠진 wheel이 세상에 나갑니다(발행 전 원격 fetch, 발행 후 wheel을 실제로 내려받아 내용 검사). PyPI는 발행이 비가역이라서, 검증의 마지막 층은 항상 **레지스트리에서 내려받은 실물**이어야 한다는 것이 우리가 얻은 결론입니다.

## 경로 3: 데이터 계약 부재 — 조회가 비결정적일 때

산출물 쪽만의 문제가 아닙니다. MCP 도구 목록에는 뜨는데 호출만 간헐적으로 404가 나던 결함의 원인은, workflow_id 컬럼에 UNIQUE 제약이 없어 같은 id로 두 행(발행본과 오래된 초안)이 공존한 것이었습니다. 목록 조회는 발행 필터로 한 행을, 호출 조회는 `limit=1`로 다른 행을 집었습니다. 라우트가 두 파일에 중복 등록돼 한쪽이 가려져 있던 결함도 같은 부류입니다 — **유일성이 보장되지 않은 식별자와 단일화되지 않은 등록 지점은 반드시 비결정성으로 돌아옵니다.** 수정은 조회 필터 일원화와 등록 지점 단일화였습니다.

## 경로 4: 설정값 검증 부재 — 한 값이 시스템 장애가 될 때

노드에 설정된 `max_tokens=1,000,000`이 모델 한계(128,000)를 넘어 매 호출이 검증 예외로 실패하고, 실행기는 90 iteration의 빈 호출로 $1.21을 소모한 뒤 실패로 끝난 장애가 있었습니다. 원인 분석에서 얻은 일반화는 두 가지였습니다 — **사용자 설정값은 실행 대상(모델)의 한계와 교차 검증해 자동으로 clamp하고, 반복 루프에는 "진전 없는 반복" 감지를 둔다.** 둘 다 엔진에 들어갔습니다.

## 정리하며

이 경로들의 공통점은 어느 것도 "코드 로직의 버그"가 아니라는 점입니다. 추적, 캐시, 경합, 계약, 검증 — 전부 코드 바깥의 구조입니다. 그래서 방어도 코드 리뷰가 아니라 층으로 쌓았습니다: fresh-clone 빌드 → 발행 전 fetch → 발행 후 실물 검사 → 산출물 프로세스 종단 검증. 다음 편은 이렇게 신뢰할 수 있게 된 실행기 위에서, 설정이 모델의 격차를 얼마나 지우는지를 통제된 벤치마크로 잰 기록입니다.

**같은 문제를 겪는 팀에게** — 우리가 겪으며 얻은 생각들입니다.
- 배포 검증을 fresh-clone 빌드 기준으로 하고 발행 후 레지스트리 실물을 내려받아 확인하는 절차를 고려해 보시면 좋을 것 같습니다. 우리 경우 로컬 빌드 성공이 저장소 상태의 증명이 아니었습니다.
- 조회 식별자의 유일성(UNIQUE)과 등록 지점의 단일성이 없으면 비결정성으로 돌아온다는 것을 겪었습니다. 계약으로 강제해 두는 편이 안전하다고 판단했습니다.
- 사용자 설정값을 실행 대상의 한계와 교차 검증(clamp)하고, 루프에 "진전 없는 반복" 감지를 두는 것도 도움이 됐습니다.
