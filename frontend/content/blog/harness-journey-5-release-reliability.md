**한 줄 요약** — 소스와 실제 배포된 산출물이 달라지는 원인은 추적 누락, 캐시 재사용, 데이터 계약 부재, 설정값 검증 부족이었습니다. 우리는 각각을 별도의 검증 단계로 막았습니다. 문제는 코드가 아니라 배포 과정에 있었기 때문입니다.

7주에 걸쳐 공개 패키지 저장소(PyPI/npm)에 30여 회 릴리즈하며, 우리는 "내 소스"와 "사용자가 설치하는 산출물"이 달라질 수 있는 원인을 하나씩 직접 겪었습니다. 이번 편은 그 원인들의 목록과, 각각에 세운 방어 단계입니다.

## 원인 1: 추적 누락 — 저장소가 소스의 전부가 아닐 때

가장 뼈아픈 사례는 버전 관리 제외 목록(`.gitignore`)이었습니다. 임시 파일을 거르려고 넣은 규칙(`_*.py`)이 파이썬 패키지의 필수 초기화 파일(`__init__.py`)까지 걸러 버려, 새 패키지의 핵심 파일이 저장소에서 계속 누락되고 있었습니다. 로컬 빌드는 로컬 환경에 남아 있던 미추적 파일 때문에 정상이지만, 저장소를 새로 내려받아(fresh clone) 빌드한 설치 파일(wheel)은 패키지 구조가 깨진 채 네 개 버전이 연속 배포됐습니다. 같은 패턴이 테스트 디렉토리도 막고 있었습니다 — 테스트가 없었던 것이 아니라, 저장소에서 제외되고 있었던 것입니다(해제 후 0→237개로 성장).

방어 단계: **배포 검증은 반드시 저장소를 새로 내려받은(fresh clone) 환경을 기준으로.** 로컬 빌드 성공은 "디스크에 있는 파일"의 증명일 뿐 "저장소에 있는 파일"의 증명이 아닙니다.

## 원인 2: 캐시와 경합 — 배포 후에도 문제가 생길 수 있습니다

핫픽스를 내도 CI(자동 빌드 시스템)의 도커 레이어 캐시가 깨진 이전 버전을 재사용할 수 있고(캐시 강제 갱신을 복구 절차에 포함), 두 작업 흐름이 같은 버전 번호로 동시에 발행하면 한쪽 수정이 빠진 설치 파일이 배포될 수 있습니다(발행 전 원격 상태 확인, 발행 후 실제로 내려받아 내용 검사). 공개 패키지 저장소는 발행을 되돌릴 수 없어서, 검증의 마지막 단계는 항상 **패키지 저장소에서 실제 내려받은 산출물**이어야 한다는 것이 우리가 얻은 결론입니다.

## 원인 3: 데이터 계약 부재 — 조회 결과가 일관되지 않을 때

산출물 쪽만의 문제가 아닙니다. 도구 목록에는 뜨는데 호출만 간헐적으로 "찾을 수 없음(404)"이 나던 결함의 원인은, 워크플로우 ID 컬럼에 중복을 금지하는 제약(UNIQUE)이 없어 같은 ID로 두 행(발행본과 오래된 초안)이 공존한 것이었습니다. 목록 조회와 호출 조회가 서로 다른 데이터를 참조하고 있었던 것입니다. 같은 API 경로가 두 파일에 중복 등록돼 한쪽이 가려져 있던 결함도 같은 부류입니다 — **유일성이 보장되지 않은 식별자와 단일화되지 않은 등록 지점은 반드시 예측할 수 없는 동작으로 이어집니다.** 수정은 조회 필터 일원화와 등록 지점 단일화였습니다.

## 원인 4: 설정값 검증 부재 — 한 값이 시스템 장애가 될 때

노드에 설정된 최대 출력 길이(`max_tokens=1,000,000`)가 모델의 한계(128,000)를 넘어 매 호출이 실패하고, 실행기는 90회의 의미 없는 반복으로 $1.21을 소모한 뒤 실패로 끝난 장애가 있었습니다. 이 문제를 분석하며 얻은 교훈은 두 가지였습니다 — **사용자 설정값은 실행 대상(모델)의 한계와 교차 검증해 한계값으로 자동 보정(clamp)하고, 반복 루프에는 "진전 없는 반복" 감지를 둔다.** 둘 다 엔진에 들어갔습니다.

## 정리하며

이 문제들의 공통점은 어느 것도 "코드 로직의 버그"가 아니라는 점입니다. 추적, 캐시, 경합, 계약, 검증 — 전부 코드 자체가 아니라 운영과 배포 구조의 문제입니다. 그래서 해결도 코드 수정이 아니라 검증 단계를 하나씩 추가하는 방식으로 접근했습니다: 새로 내려받은 저장소 기준 빌드 → 발행 전 원격 상태 확인 → 발행 후 실제 산출물 검사 → 산출물 프로세스 종단 검증. 다음 편은 이렇게 신뢰할 수 있게 된 실행기 위에서, 설정이 모델의 격차를 얼마나 지우는지를 통제된 벤치마크로 검증한 이야기입니다.

**같은 문제를 겪는 팀에게** — 우리가 겪으며 얻은 생각들입니다.
- 배포 검증을 "저장소를 새로 내려받아 빌드" 기준으로 하고, 발행 후에는 패키지 저장소의 실제 산출물을 내려받아 확인하는 절차를 한 번쯤 검토해 볼 만합니다. 우리 경우 로컬 빌드 성공이 저장소 상태의 증명이 아니었습니다.
- 조회 식별자의 유일성(UNIQUE)과 등록 지점의 단일성이 없으면 예측할 수 없는 동작으로 이어진다는 것을 겪었습니다. 계약으로 강제해 두는 편이 안전하다고 판단했습니다.
- 사용자 설정값을 실행 대상의 한계와 교차 검증(clamp)하고, 루프에 "진전 없는 반복" 감지를 두는 것도 실제로 장애를 줄이는 데 효과적이었습니다.
